# ADR-38911: Emit AWF chroot config via runtime in-place patch for ARC/DinD

**Date**: 2026-06-12
**Status**: Draft

## Context

ARC (Actions Runner Controller) runners using Docker-in-Docker (DinD) have a split filesystem topology: the runner pod and the Docker daemon do not share a filesystem. Historically, gh-aw required a separate bootstrap action to copy the agent binary (e.g., `copilot`) into the daemon-visible filesystem and to pre-seed `/etc/passwd` with the runner's identity. AWF v0.27.1+ now supports `chroot.binariesSourcePath` and `chroot.identity.*` natively, which lets AWF stage binaries and resolve identity without that bootstrap action. The blocker was that gh-aw's config builder never emitted these fields. A key constraint is that the runner's identity (`uid`, `gid`, `user`) is only knowable at job runtime, not at workflow-compile time, and the generated run step must stay within GitHub Actions' 21 KB per-step expression limit.

## Decision

We will emit `chroot.binariesSourcePath` and `chroot.identity.*` by patching the AWF config JSON **at runtime**, rather than at compile time. When the selected AWF version is ≥ v0.27.1 (`AWFChrootConfigMinVersion`) and DinD topology is detected (via `DOCKER_HOST` matching `^tcp://`), a repository JavaScript helper invoked from the generated run step reads `${RUNNER_TEMP}/gh-aw/awf-config.json`, injects the `chroot` object — with static `binariesSourcePath`/`home` constants and runtime-resolved identity values from the Node runtime — and writes it back in place before AWF reads it. The patch body is merged inside the existing `arcDindPrefixProbe` `DOCKER_HOST` `if`-block so the script contains only one `DOCKER_HOST` condition check. Corresponding Go structs (`AWFChrootConfig`, `AWFChrootIdentityConfig`) are added for schema completeness only.

## Alternatives Considered

### Alternative 1: Keep the existing bootstrap action
Continue requiring ARC/DinD users to add a bootstrap action that copies the binary and pre-seeds `/etc/passwd`. Rejected because it pushes setup complexity onto every workflow author, is error-prone, and duplicates capability AWF now provides natively. The whole point of the AWF v0.27.1 feature is to remove this action.

### Alternative 2: Populate `chroot` at compile time in `BuildAWFConfigJSON`
Emit the `chroot` object directly when the Go config file is generated, alongside the other config sections. Rejected because the runner identity (`uid`/`gid`/`user`) is not known when the workflow YAML is compiled — it is a property of the runtime environment. Hard-coding placeholder identity values would be incorrect on real ARC/DinD runners, which is exactly the failure mode this change avoids.

## Consequences

### Positive
- ARC/DinD users no longer need a bootstrap action to copy binaries or pre-seed `/etc/passwd`.
- Identity is resolved from the actual runtime environment (`id -un/-u/-g`), so it is always correct for the runner the job lands on.
- The change is version-gated and topology-gated, so non-DinD and older-AWF setups are unaffected.

### Negative
- Introduces a runtime dependency on the repository JavaScript helper and Node.js inside the generated run step.
- Config is now assembled in two places (compile-time Go builder + runtime helper), so the `chroot` shape must be kept in sync across the Go structs and the embedded script by hand.

### Neutral
- All engine golden files were regenerated to include the runtime helper patch invocation.
- A standalone `buildArcDindChrootConfigInjectScript` form exists purely for test coverage; production uses the merged in-`if`-block form.
