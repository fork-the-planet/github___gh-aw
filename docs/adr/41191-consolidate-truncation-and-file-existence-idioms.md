# ADR-41191: Consolidate inline truncation and file-existence idioms into shared utilities

**Date**: 2026-06-24
**Status**: Draft

## Context

The `pkg/cli` and `pkg/workflow` packages had accumulated two recurring hand-rolled idioms. First, roughly nine sites reimplemented string truncation inline (`if len(s) > N { s = s[:N-3] + "..." }`), each with its own ellipsis length and off-by-three convention — for example `logs_download.go` used a `maxLen 203` form. Second, around fifty sites performed pure boolean existence checks via `if _, err := os.Stat(path); err == nil`, mixing file and directory semantics without expressing intent. The repository already ships canonical helpers — `stringutil.Truncate` and `fileutil.FileExists`/`fileutil.DirExists` — so the duplicated idioms were inconsistent and easy to get subtly wrong. This PR is a mechanical refactor that does not change runtime behavior, only the call sites.

## Decision

We will replace inline string-truncation expressions with `stringutil.Truncate`, and replace boolean-only `os.Stat` existence checks with `fileutil.FileExists` / `fileutil.DirExists`. The driver is consistency and a single source of truth for two error-prone idioms. Sites that need richer `os.Stat` semantics — `FileInfo` inspection, explicit `os.IsNotExist` handling, error propagation, or ambiguous file-or-directory paths — are deliberately left untouched.

## Alternatives Considered

### Alternative 1: Leave the inline idioms in place

The inline forms already work and are individually trivial. Leaving them avoids touching ~35 files and any associated review/merge-conflict cost. Rejected because the duplication had already produced inconsistencies (varying ellipsis conventions, file-vs-directory confusion) and each new occurrence is one more place to get the off-by-three or `os.IsNotExist` semantics wrong.

### Alternative 2: Add a linter to flag the idioms instead of migrating

The repo maintains a custom `pkg/linters` analyzer suite, so a rule could flag inline truncation and boolean `os.Stat` checks rather than rewriting them now. Rejected as the sole approach because a linter only prevents future occurrences and still leaves the existing ~60 sites unmigrated; a follow-up linter could complement this change but does not replace it. *(Not inferable from this PR whether such a linter is planned.)*

## Consequences

### Positive

- Single source of truth: truncation and existence semantics are defined once in `stringutil` / `fileutil`, so a fix or behavior change applies everywhere.
- Call sites read by intent (`fileutil.DirExists(dir)` vs `os.Stat` + `err == nil`), and several files drop their direct `os` import.

### Negative

- Adds import coupling from many `pkg/cli` / `pkg/workflow` files to `fileutil` and `stringutil`.
- Behavior-equivalence depends on the helpers exactly matching each prior inline convention — notably the off-by-three `maxLen 203` truncation in `logs_download.go`; a mismatch would silently change user-facing output. *(Reviewer should confirm `stringutil.Truncate` reproduces each replaced convention.)*

### Neutral

- The migration is intentionally partial: sites using `FileInfo`, `os.IsNotExist`, error propagation, or ambiguous paths are skipped, so both idiom styles still coexist in the codebase.
- Some debug/verbose log lines lost the underlying `err` value (e.g. `"No aw_info.json found at %s"` no longer prints `%v`), a minor reduction in diagnostic detail.
