# ADR-38725: Resolve Copilot Config Paths from `$HOME` Instead of a Hard-Coded `/home/runner`

**Date**: 2026-06-12
**Status**: Draft

## Context

The Copilot engine generator hard-coded `/home/runner/.copilot` (and `XDG_CONFIG_HOME=/home/runner`) in several places in the Go compiler and runtime scripts. This breaks workflows on self-hosted, containerized, or rootless runners whose `$HOME` is not `/home/runner` (e.g. `/home/actions`, `/root`, `/var/lib/runner`), where the Copilot CLI either fails to find its config or writes it to the wrong location. Two of the affected paths were emitted into the YAML `env:` block, where GitHub Actions does **not** shell-expand values — so any `$HOME`-based value there would reach the agent as the literal string `$HOME/...`. The fix therefore requires both swapping the literal path for `$HOME` and relocating certain variables out of YAML `env:` into the `run:` script where the shell performs expansion. The change must remain a no-op on standard GitHub-hosted runners (`HOME=/home/runner`).

## Decision

We will resolve all Copilot configuration paths from the `$HOME` environment variable rather than a hard-coded `/home/runner`, and we will emit the affected exports (`XDG_CONFIG_HOME`, `GH_AW_MCP_CONFIG`) from the generated `run:` shell script rather than the YAML `env:` block so the shell expands `$HOME` at execution time. We will use bare `$HOME` with no fallback in shell-expanded contexts (consistent with the project's other generators), while the JS and `.sh` helpers validate that `HOME` is set before writing a Copilot path so an unset `HOME` fails loudly instead of writing to `/.copilot`. The cleanup trap body is single-quoted so `$HOME` expands at trap-fire time, not trap-definition time. The change is scoped to the `copilot` engine; the shared MCP gateway only consults `HOME` when actually writing the Copilot fallback config, so non-Copilot engines gain no new `HOME` dependency.

## Alternatives Considered

### Alternative 1: Use a fallback default `${HOME:-/home/runner}`
This would preserve the current behavior when `HOME` is unset. It was rejected because the project's other generators (`mcp_setup_generator.go`, the codex engine config, the session-state copy script) already rely on bare `$HOME` with no fallback; adding a fallback only here would be inconsistent, and a genuinely unset `HOME` indicates a deeper environment problem better surfaced as a loud failure than masked by a silent wrong-path write.

### Alternative 2: Keep paths in YAML `env:` and resolve at compile time
Resolving the runner's home directory when the workflow is compiled would avoid touching the `run:` script. It was rejected because the compiler runs on a different machine/context than the eventual runner, so the home directory is not known at compile time; the value must be resolved at execution time by the shell, which YAML `env:` cannot do.

## Consequences

### Positive
- Copilot workflows function correctly on self-hosted, containerized, and rootless runners with non-default `$HOME` values.
- Failures from an unset `HOME` are explicit (JS/`.sh` helpers throw) rather than silently writing config to `/.copilot`.

### Negative
- Some configuration moves from declarative YAML `env:` into imperative `run:` shell, making the generated workflow slightly harder to read at a glance.
- All 245 generated `.lock.yml` files were recompiled, producing large mechanical diffs; downstream user repos will see similar diffs on their next `gh aw compile`.

### Neutral
- Behavior on standard GitHub-hosted runners (`HOME=/home/runner`) is unchanged.
- Two `.sh` scripts not currently invoked by the generated workflow were updated for consistency with their `.cjs` counterparts.
- No codemod is needed; the affected text appears only in compiler output, not user-authored markdown.
